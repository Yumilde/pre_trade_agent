# prompt_rewriter 规则库

本目录存放 `prompt_rewriter` 子代理使用的「问题重写与歧义检测」规则。规则以 JSON 文件外置维护，**构建 Agent 时由 `agent.py` 的 `_load_rules()` 读取并注入子代理的 system_prompt**（不上传到 Daytona 沙箱，与 `/formulas/`、`/scenarios/` 无关）。新增 / 修改规则只需编辑本目录下的 JSON，下次构建即生效。

## 目录结构

```
rules/
├── commonsense/   # 信用账户常识语义规则，一条规则一个 .json
├── dimensions/    # 四维歧义检测定义，一个维度一个 .json
└── README.md      # 本文件
```

## commonsense/<rule_id>.json —— 常识语义规则

用于捕捉融资融券行业「言外之意」（如「提取资金」=卖券释放保证金而非存钱）。

- `rule_id`：规则唯一标识
- `patterns`：触发该规则的关键词/正则模式列表（命中任一即可视为相关）
- `dimension`：该规则归属的歧义维度（目标动作 / 计算口径 / 作用对象 / 约束优先级）
- `implication`：命中后应推断出的隐含含义（动作 / 口径指标 / 约束）
- `confidence`：推断置信度（high / medium / low）
- `reasoning`：推断依据（行业惯例 / 常见语义 / 口径说明）
- `notes`：补充信息（公式、典型工作流、口径风险等，可为空对象）

## dimensions/<dimension>.json —— 四维歧义定义

用于检测某维度下问题是否同时指向多个互斥方向（即存在歧义）。

- `dimension`：维度中文名
- `description`：维度说明
- `directions`：该维度下的方向组列表，每项含：
  - `group`：方向组标识
  - `label`：展示给用户的中文标签
  - `keywords`：触发该方向组的关键词/正则模式列表
- `special_handling`：该维度的特殊判定说明（无则为 null）

## 子代理使用约定（保守口径 + 开放判定 + 迭代重写）

角色定位（重要）：
- `commonsense/` = **固化负向清单**：机构口径既定事实（如「提取资金」=卖券、卖券还款现金不变），命中即直接消解并在重写中标注隐含推断，**严禁据此 [CLARIFY]**。
- `dimensions/` = **非穷尽歧义检查单**：仅作启发，不是封闭闸门；真歧义以"判定总原则"为准，可识别清单之外的任何此类方面。

判定总原则（保守）：仅当**同时**满足——(a) 存在 ≥2 种都成立、且会改变数值结果/建议动作的解读或取值；(b) 该选择属用户意图/偏好，或只有用户/外部才知道的信息；(c) 无法用固化负向清单或行业默认主方向消解——才发起 [CLARIFY]；能定下来的绝不反问。

流程：
1. 对照 `commonsense/` 识别隐含语义，按固化事实直接消解、写入重写。
2. 按总原则判断是否存在"需用户拍板"的事项：既含真歧义（`dimensions/` 仅作启发、不限于这些维度，遵守各维度 `special_handling`），也含**影响结果却缺失/不可得的信息**（如某账户量查询失败、用户独有阈值未知）——后者应**先问用户**而非直接终止。
3. 无残留待定事项且无互相矛盾的强推断 → 输出（最新版）重写；否则 → 输出 `[CLARIFY]`：**无论歧义还是缺信息都必须给具体选项**（A/B/C…，高置信首项标「（推荐）」，每项附「其他（请说明）」；缺信息类尽量附常见取值候选）。
4. 迭代重写：携完整澄清历史（原始问题 + 历次 [CLARIFY] + 用户历次回答）；用户已回答的事项不再追问，且只要其回答对最近一版重写有补充/澄清就**再重写一次**，以最新重写为准。
5. 澄清预算：同一轮对话至多 4 轮（运行时注入剩余轮次提示）；达上限仍无法消解则不再 [CLARIFY]，交主代理按 [STOP]（unresolved_ambiguity / missing_critical_input）收口。
6. 放权语义：用户表达「按你判断 / 你定 / 按默认 / 按行业惯例」时，全部按推荐主方向消解、不再追问。
